# Safe-OS 最小实验运行和指标信息

本文只整理当前 Safe-OS 这条实验链路本身，不讨论 `autoresearch` 目录下的框架示例文件。目标是先把“最小实验怎么稳定跑起来、会产出什么、该看哪些指标”讲清楚，方便后续再改训练脚本和优化实验设计。

---

## 一、当前实验链路的真实入口

当前最小实验的主入口是：

- 训练脚本：`/data/Agent_Defense/code/SQUIRL/scripts/evolve_train.py`
- Phase 1 初始技能库：`/data/Agent_Defense/code/SQUIRL/runs/run_v5_full_mass_new/skills_evolved/`
- Safe-OS benign 数据：`/data/AGrail4Agent/DAS/data/safe-os/benign.json`
- Agent-SafetyBench unsafe 训练数据：`/data/Agent_Defense/code/Agent-SafetyBench-main/data/released_data_train.json`

当前这条链路里，`evolve_train.py` 负责：

1. 逐条读取训练样本。
2. 让 conductor 产生工具调用。
3. 用 SQUIRL 判断这次调用是否应拦截。
4. 用 `RewardEngine` 判断这次调用“实际上是否安全”。
5. 按 `TP / FP / FN / TN` 归类。
6. 触发 V5 skill 的新增、强化或放松。
7. 落盘日志、checkpoint 和演化后的技能库。

---

## 二、最小实验的目标

这里说的“最小实验”，不是为了追求最好效果，而是为了最低成本验证下面 4 件事同时成立：

1. 数据能被正常读入。
2. `evolve_train.py` 能完整跑完一轮。
3. 日志、checkpoint、skills 输出正常。
4. 训练结果里同时出现 unsafe 和 benign，两类样本都能进入指标统计。

推荐把“最小实验”定义为：

- 使用现成的 Phase 1 skill 库；
- 使用 Safe-OS benign + Agent-SafetyBench unsafe 的混合数据；
- 开启 `--v5`；
- 跑一个较小的 `max_samples`，先验证链路和指标，再决定是否扩大规模。

---

## 三、一个必须先说明的现状问题

当前 `evolve_train.py` 的行为有一个对“最小实验”非常重要的影响：

1. 它先执行 `data = data[:max_samples]`。
2. 然后才统计 safe / unsafe 比例并尝试做 benign oversample。

这意味着：

- 如果合并后的数据文件前面全是 unsafe，而 benign 在文件尾部，
- 那么当 `max_samples` 很小的时候，
- 训练切片里可能一个 benign 都没有。

当前文档里原来的简单拼接方式：

```python
combined = u + b
```

在最小实验场景下不够好，因为前 `N` 条很可能全部是 unsafe。  
所以最小实验不建议直接用“unsafe 全在前、benign 全在后”的拼接文件，而应该额外准备一个“前段就能覆盖 benign”的最小实验数据文件。

---

## 四、数据准备

### 4.1 数据现状

- `released_data_train.json` 顶层没有可直接用的 `labels=1` 字段，需要手工补。
- `benign.json` 的 `create/init` 结构并不完全同构，转换脚本要写得更稳，不能默认 `init` 一定是字典。
- 当前数据规模：
  - unsafe：1600 条
  - benign：27 条
  - 理论合并总数：1627 条

### 4.2 生成两个数据文件

建议一次性生成两个文件：

- `/tmp/combined_safeos_full.json`
  - 用于正式跑全量或大样本实验。
- `/tmp/combined_safeos_minimal.json`
  - 用于最小实验。
  - 特意把 benign 交错插入前段，避免小样本时完全看不到 benign。

执行命令：

```bash
python3 - <<'PY'
import json
from pathlib import Path

benign_path = Path('/data/AGrail4Agent/DAS/data/safe-os/benign.json')
unsafe_path = Path('/data/Agent_Defense/code/Agent-SafetyBench-main/data/released_data_train.json')

benign_raw = json.load(open(benign_path))
unsafe_raw = json.load(open(unsafe_path))

benign = []
for idx, item in enumerate(benign_raw):
    create_data = item.get('create')
    if isinstance(create_data, str):
        env_name = 'default'
        init_code = create_data
    else:
        env_name = create_data.get('local', 'default')
        init_field = create_data.get('init', {})
        if isinstance(init_field, dict):
            init_code = init_field.get('code', '')
        else:
            init_code = str(init_field)

    benign.append({
        'id': f'benign_{idx}',
        'instruction': item.get('description', ''),
        'labels': 0,
        'risks': 'This is a benign request.',
        'environments': [{
            'name': env_name,
            'parameters': {'code': init_code},
            'tools': ['*'],
        }],
    })

unsafe = []
for idx, item in enumerate(unsafe_raw):
    x = dict(item)
    x['labels'] = 1
    unsafe.append(x)

# full 文件：保留完整数据
combined_full = unsafe + benign

# minimal 文件：把 benign 尽量插到前面，保证小样本时也能覆盖
combined_minimal = []
unsafe_idx = 0
for b in benign:
    take = unsafe[unsafe_idx:unsafe_idx + 3]
    combined_minimal.extend(take)
    combined_minimal.append(b)
    unsafe_idx += len(take)
combined_minimal.extend(unsafe[unsafe_idx:])

with open('/tmp/combined_safeos_full.json', 'w') as f:
    json.dump(combined_full, f, ensure_ascii=False, indent=2)

with open('/tmp/combined_safeos_minimal.json', 'w') as f:
    json.dump(combined_minimal, f, ensure_ascii=False, indent=2)

print('unsafe =', len(unsafe))
print('benign =', len(benign))
print('combined_full =', len(combined_full))
print('combined_minimal =', len(combined_minimal))
print('first_20_safe_in_minimal =', sum(1 for x in combined_minimal[:20] if x.get('labels') == 0))
PY
```

### 4.3 为什么要单独做 `combined_safeos_minimal.json`

因为当前脚本先截断 `max_samples`，再做 balance。  
如果最小实验直接读 `unsafe + benign`，前 20、50、100 条都可能只有 unsafe。

用上面的交错文件后：

- `max_samples=20` 时，前 20 条里会稳定出现 benign；
- 可以更早看到 `FP / TN / benign_learned`；
- 对后续优化更有帮助。

## 五、最小实验如何运行

### 5.1 推荐的最小实验命令

建议先跑一个新的输出目录，不要复用已有目录：

```bash
cd /data/Agent_Defense/code
conda activate AGrail

python -m SQUIRL.scripts.evolve_train \
    --train_data "/tmp/combined_safeos_minimal.json" \
    --skills_db "SQUIRL/runs/run_v5_full_mass_new/skills_evolved" \
    --output_path "SQUIRL/runs/safeos_minimal_v1" \
    --model "gpt-5.2" \
    --max_samples 20 \
    --checkpoint_every 10 \
    --v5 \
    --no_resume
```

### 5.2 这个最小实验的含义

- `--train_data /tmp/combined_safeos_minimal.json`
  - 让小样本前段就包含 benign。
- `--output_path SQUIRL/runs/safeos_minimal_v1`
  - 单独落盘，避免和已有 run 混在一起。
- `--max_samples 20`
  - 先验证链路是否稳定，成本最低。
- `--checkpoint_every 10`
  - 10 条样本就落一次 checkpoint，便于快速确认输出。
- `--v5`
  - 走当前主要使用的 V5 skill 演化路径。
- `--no_resume`
  - 明确从头跑，不读旧 checkpoint。

### 5.3 建议的实验梯度

可以按下面的节奏逐步放大：

1. `max_samples=20`：只验证能不能跑通，日志和指标是否正常。
2. `max_samples=50`：开始观察 `FP / FN / TN / TP` 是否有代表性。
3. `max_samples=100`：开始有一定统计意义。
4. `max_samples=200`：接近当前已有 smoke test 规模。

正式实验可以改为：

```bash
python -m SQUIRL.scripts.evolve_train \
    --train_data "/tmp/combined_safeos_full.json" \
    --skills_db "SQUIRL/runs/run_v5_full_mass_new/skills_evolved" \
    --output_path "SQUIRL/runs/safeos_smoketest_v6" \
    --model "gpt-5.2" \
    --max_samples 200 \
    --checkpoint_every 50 \
    --v5 \
    --no_resume
```

## 七、当前训练里“真正的指标”是什么

### 7.1 最重要的事实

当前 `evolve_train.py` 里的核心判断不是直接看数据集的 `labels`，而是看两件事：

1. `squirl_blocked`
   - SQUIRL 有没有拦截这次工具调用。
2. `actually_safe`
   - `RewardEngine.evaluate(...)` 判断这次调用实际上是否安全。

也就是说：

- `labels` 当前主要用于 safe / unsafe 数据分布和 benign balance；
- 真正进入 `TP / FP / FN / TN` 统计的，是 `RewardEngine` 的判定结果。

这点对后续优化非常重要，因为“数据标签”和“在线评估判定”并不是同一层。

### 7.2 四类基础统计

训练脚本中的定义如下：

- `TP`：unsafe 且被正确拦截
  - `not actually_safe and squirl_blocked`
- `FP`：safe 但被错误拦截
  - `actually_safe and squirl_blocked`
- `FN`：unsafe 但没拦截住
  - `not actually_safe and not squirl_blocked`
- `TN`：safe 且被正确放行
  - `actually_safe and not squirl_blocked`

在这个实验里，“正类”可以理解为“应该拦截的危险行为”。

### 7.3 其它直接记录的统计量

脚本还会直接累计这些字段：

- `processed`
  - 进入主循环并尝试处理的样本数。
- `skipped`
  - 因环境解析失败、没有 tool call 等原因被跳过的样本数。
- `failures`
  - `FP + FN`。
- `updates`
  - 触发 skill 新增/演化/放松的总次数。
- `benign_learned`
  - 因 benign / safe 样本触发“放松 skill”而发生的更新次数。
- `Retrieval Hit Rate`
  - 检索命中率；日志里会按 `retrieval_hits / retrieval_total` 输出百分比。
- `Final Skills Count`
  - 普通 skill 数量。
- `Final V5 Skills Count`
  - V5 skill 数量。

## 八、建议关注的派生指标

训练脚本本身没有直接打印 `precision / recall / f1 / specificity / accuracy`，但可以根据 `TP / FP / FN / TN` 自己算。

设：

```text
TP = true_positive
FP = false_positive
FN = false_negative
TN = true_negative
N_eval = TP + FP + FN + TN
```

则建议至少看下面这些：

### 8.1 拦截精度

```text
Precision = TP / (TP + FP)
```

含义：

- 被拦截的调用里，真正危险的比例。
- 越高越说明“少误杀”。

### 8.2 拦截召回

```text
Recall = TP / (TP + FN)
```

含义：

- 所有真正危险的调用里，被拦住的比例。
- 越高越说明“少漏拦”。

### 8.3 F1

```text
F1 = 2 * Precision * Recall / (Precision + Recall)
```

含义：

- 同时综合 Precision 和 Recall。
- 后续做优化时，通常它比只看单一指标更稳。

### 8.4 Specificity

```text
Specificity = TN / (TN + FP)
```

含义：

- 对安全请求的放行能力。
- 越低说明误杀越严重。

### 8.5 Accuracy

```text
Accuracy = (TP + TN) / (TP + FP + FN + TN)
```

含义：

- 在真正参与判定的样本上，整体判断正确率。

### 8.6 Skip Rate

```text
Skip Rate = skipped / processed
```

含义：

- 样本里有多少根本没进入有效判定。
- 如果这个值高，说明当前实验链路稳定性还不够。

### 8.7 Failure Rate

```text
Failure Rate = (FP + FN) / (TP + FP + FN + TN)
```

含义：

- 把误杀和漏拦一起看。
- 越低越好。

## 九、为什么 `processed` 不能直接当样本总数

当前脚本里：

- `processed` 在样本刚进入循环时就先 `+1`；
- 但后面仍可能因为环境解析失败、没有有效 tool call 等原因进入 `skipped`。

所以：

- `processed` 是“尝试处理过”；
- `TP + FP + FN + TN` 才是“真正进入判定矩阵”的样本数。

后续做指标对比时，建议始终同时报告：

1. `processed`
2. `skipped`
3. `N_eval = TP + FP + FN + TN`

不然不同实验之间可能看起来都跑了 100 条，但真正进入判定的数量完全不同。

## 十、跑完后怎么取指标

### 10.1 直接看日志末尾

```bash
tail -n 50 /data/Agent_Defense/code/SQUIRL/runs/safeos_minimal_v1/train.log
```

### 10.2 直接看 checkpoint 汇总

```bash
cat /data/Agent_Defense/code/SQUIRL/runs/safeos_minimal_v1/checkpoints/progress.json
```

### 10.3 用一个小脚本计算派生指标

```bash
python3 - <<'PY'
import json

path = '/data/Agent_Defense/code/SQUIRL/runs/safeos_minimal_v1/checkpoints/progress.json'
data = json.load(open(path))
s = data['stats']

tp = s['true_positive']
fp = s['false_positive']
fn = s['false_negative']
tn = s['true_negative']
processed = s['processed']
skipped = s['skipped']
evaluated = tp + fp + fn + tn

def ratio(a, b):
    return a / b if b else 0.0

precision = ratio(tp, tp + fp)
recall = ratio(tp, tp + fn)
f1 = ratio(2 * precision * recall, precision + recall)
specificity = ratio(tn, tn + fp)
accuracy = ratio(tp + tn, evaluated)
skip_rate = ratio(skipped, processed)
failure_rate = ratio(fp + fn, evaluated)

print('processed =', processed)
print('skipped =', skipped)
print('evaluated =', evaluated)
print('TP FP FN TN =', tp, fp, fn, tn)
print('precision =', round(precision, 4))
print('recall =', round(recall, 4))
print('f1 =', round(f1, 4))
print('specificity =', round(specificity, 4))
print('accuracy =', round(accuracy, 4))
print('skip_rate =', round(skip_rate, 4))
print('failure_rate =', round(failure_rate, 4))
print('updates =', s['updates'])
print('benign_learned =', s['benign_learned'])
PY
```

## 十一、已有一次运行可作为参考

当前目录下已有一份运行结果：

- `SQUIRL/runs/safeos_smoketest_v5/checkpoints/progress.json`

其中 step 100 的统计是：

```text
processed = 100
skipped = 36
TP = 30
FP = 32
FN = 1
TN = 1
updates = 70
benign_learned = 17
```

据此可得：

- `evaluated = 64`
- `precision = 30 / (30 + 32) = 0.4839`
- `recall = 30 / (30 + 1) = 0.9677`
- `f1 = 0.6452`
- `specificity = 1 / (1 + 32) = 0.0303`
- `accuracy = (30 + 1) / 64 = 0.4844`

这组结果说明当前模型在这个阶段：

- 对危险行为的召回已经很高；
- 但对 benign / safe 请求的误杀仍然非常严重；
- 后续优化重点应该是降低 `FP`、提高 `TN` 和 `specificity`。

## 十二、后续优化前，建议先固定的最小观测面板

从下一轮开始，建议每次实验至少固定记录这 10 个值：

1. `processed`
2. `skipped`
3. `TP`
4. `FP`
5. `FN`
6. `TN`
7. `precision`
8. `recall`
9. `f1`
10. `specificity`

如果再多加两个，我建议补上：

11. `updates`
12. `benign_learned`

原因很简单：

- `TP / FN / recall` 看的是“拦不拦得住”；
- `FP / TN / specificity` 看的是“会不会误杀”；
- `updates / benign_learned` 看的是 skill 演化是否在真正吸收 benign 反馈。

## 十三、结论

当前最小实验的推荐做法是：

1. 先生成 `combined_safeos_minimal.json`，不要直接用简单拼接版。
2. 用一个全新的 `output_path` 跑 `max_samples=20` 的 V5 实验。
3. 跑完先看 `progress.json` 和 `train.log`。
4. 不只看 `TP / FN`，必须同时看 `FP / TN / specificity`。

就当前实现来看，后续优化前最值得优先盯住的是两件事：

1. 小样本时 benign 是否真的进入了训练切片。
2. `RewardEngine` 判定下的 `FP` 是否能明显下降。

本文基于以下材料整理当前最小实验的优化目标：

  - `/data/Agent_Defense/code/SQUIRL/runs/safeos_smoketest_v3/`
  - `/data/Agent_Defense/code/SQUIRL/runs/safeos_smoketest_v5/`

  - `/data/Agent_Defense/Discussion.md`
- 研究蓝图：

## 一、先给结论

当前最小实验的核心优化目标，不是继续扩展新 benchmark，也不是追求完整论文结果，而是先把下面这件事做扎实：

**在保持 unsafe 拦截能力基本不退化的前提下，让 benign 样本在小样本实验中被真正看到、真正学习、并真实降低误杀。**

换成更具体的话，就是：

1. 修通 benign supervision 在最小实验里的有效覆盖。
2. 把当前最突出的 `FP` 问题压下去。
3. 同时守住 `unsafe recall`，避免为了降误杀而明显放松防御。
4. 让最小实验的指标能稳定反映真实趋势，而不是被数据切片和 skipped 样本污染。

---

## 二、现有运行记录说明了什么

### 1. 历史 full run 证明过“监督链路曾经是坏的”

`run_v5_full_mass_new/checkpoints/progress.json` 的结果是：

- `TP=0`
- `FP=0`
- `FN=541`
- `TN=493`

这说明那一阶段的训练几乎没有形成有效的“该拦截就拦截”的正向学习，和 `Discussion.md` 里提到的历史问题一致：之前的 ground truth / benign supervision 链路是有问题的。

这部分不是当前最小实验的直接目标，但它决定了当前最小实验首先必须验证“监督链路已经真的修好”。

### 2. 现有 smoke test 的主要改进，只发生在“开始学 benign”这一层

已有几个关键 run：

| Run | Processed | Skipped | TP | FP | FN | TN | Benign Learned |
|---|---:|---:|---:|---:|---:|---:|---:|
| `safeos_smoketest_v2` | 50 | 18 | 15 | 17 | 0 | 0 | 0 |
| `safeos_smoketest_v3` | 100 | 38 | 26 | 35 | 1 | 0 | 0 |
| `safeos_smoketest_v5` | 100 | 36 | 30 | 32 | 1 | 1 | 17 |

这里最重要的变化是：

- `v2 / v3` 里 `Benign Learned=0`，`TN=0`
- `v5` 开始出现 `Benign Learned=17`，并首次出现 `TN=1`

这说明：

- benign 学习链路开始被触发了；
- 但效果还远不够，当前只是“从完全没学到，变成勉强开始学到”。

### 3. 当前主矛盾不是 recall，而是 over-blocking

`safeos_smoketest_v5` 的派生指标大致是：

- `precision = 0.4839`
- `recall = 0.9677`
- `specificity = 0.0303`
- `accuracy = 0.4844`

这组数很明确：

- `unsafe recall` 已经很高；
- 真正拖后腿的是 `specificity` 极低；
- 也就是 safe / benign 请求仍然被大量误杀。

因此当前最小实验的优化重心应该放在：

**降低 FP、增加 TN、提高 specificity。**

而不是优先继续推高 recall。

### 4. 最小实验当前还受数据切片方式干扰

`safeos_smoketest_v5/train.log` 里能看到：

```text
Dataset loaded. Total: 1626, Training on: 200
Original dataset balance: 0 safe / 200 unsafe
[BenignBalance] Final training set: 0 safe / 200 unsafe
```

这说明在当前脚本逻辑下：

- 先做 `data[:max_samples]`
- 再做 benign balance

结果就是小样本实验前段根本没看到 benign。  
这会直接污染最小实验的判断，因为你以为自己在测“mixed smoke test”，实际上测到的还是“unsafe-only 前缀”。

### 5. skipped 比例仍然偏高，影响最小实验解释力

现有 smoke test 的 skipped 情况：

- `v2`: `18/50`
- `v3`: `38/100`
- `v5`: `36/100`

也就是说，约三分之一样本没有真正进入 `TP/FP/FN/TN` 判定矩阵。  
如果这个比例不降下来，最小实验的波动会很大，很多指标会被“没测到”而不是“测差了”影响。

## 三、结合 Discussion.md 后，当前最小实验的正式优化目标

`Discussion.md` 已经把优先级说得很清楚：  
当前最高优先级是“先完成 benign 数据链路修复后的 smoke test 验证，再决定是否扩大训练”。

据此，当前最小实验的优化目标应分成 4 个层级。

### Goal 1：确保 benign 在最小实验中被真正覆盖并触发学习

这是第一优先级，也是当前所有后续优化的前提。

具体目标：

- 最小实验的数据切片前段必须稳定包含 benign。
- `Benign Learned` 必须大于 0。
- `TN` 不能继续长期为 0。
- benign 样本不能继续呈现“统一拦截、统一不学习”的状态。

这对应 `Discussion.md` 里的主线问题：

- “先确认 benign 学习链路已经真正打通”
- “先完成 50 样本 smoke test，再决定是否进入完整训练”

### Goal 2：在不明显损失 unsafe 防御的前提下，优先压低 FP

结合现有记录，当前最值得优化的不是 `FN`，而是 `FP`。

具体目标：

- 让 `FP` 明显下降；
- 让 `TN` 和 `specificity` 明显上升；
- 同时保持 `TP` 和 `recall` 基本稳定。

这里建议直接沿用 `Discussion.md` 里的约束：

- 如果 unsafe 核心指标相对当前基线下降超过 `5%`，这一轮最小实验不算通过。

也就是说，当前最小实验追求的是：

**在 recall 基本守住的情况下，做更好的 safety-utility calibration。**

### Goal 3：让最小实验从“可运行”升级为“可解释”

当前最小实验还有一个问题：很多样本被 skip，导致结果不稳。

所以第三层目标是：

- 降低 `Skip Rate`；
- 提高真正进入 `TP/FP/FN/TN` 的样本比例；
- 让最小实验能稳定复现趋势，而不是每次只是在不同的有效样本子集上波动。

这一步虽然不是最 headline 的优化，但它直接决定：

- 你后面做的阈值调整是不是可信；
- 你看到的 `FP` 下降是不是因为样本没进来。

### Goal 4：让 skill 更新开始对“误杀修复”有可验证贡献

当前 `v5` 里虽然已经出现 `Benign Learned=17`，但指标层面的收益还非常有限。  
这意味着当前 skill relaxation 虽然发生了，但还没有充分转化成行为层面的 utility 改善。

所以第四层目标是：

- `updates` 不能只是增长；
- `benign_learned` 不能只是计数增加；
- 它们要和 `FP` 下降、`TN` 上升形成一致趋势。

也就是说，下一步要观察的不只是“有没有学”，而是：

**学到的 benign knowledge 是否真的改变了 guard decision。**

## 四、结合 idea.md 后，当前最小实验在研究蓝图中的定位

`idea.md` 里的研究问题是：

- `RQ1`: unsafe 防御效果
- `RQ2`: utility / 减少 over-blocking
- `RQ3`: 泛化
- `RQ4`: 运行开销

从当前阶段看，最小实验主要服务的是：

### 1. 当前最小实验直接服务 `RQ2`

即：

**能否在提升 malicious defense 的同时，避免对 benign 请求的过度拦截。**

现有 run 已经说明：

- unsafe recall 不是当前最大短板；
- over-blocking 才是。

所以最小实验的优化目标，本质上是先把 `RQ2` 做实。

### 2. 当前最小实验只部分服务 `RQ1`

因为 `RQ1` 的前提不是“纯粹把 recall 拉高”，而是要在真实 runtime defense 场景里形成稳定有效的 unsafe 拦截。

当前最小实验对 `RQ1` 的要求是：

- 不让 recall 明显回退；
- 不破坏已有的 unsafe 防御能力。

但它不是当前最需要突破的方向。

### 3. 当前最小实验暂时不应承担 `RQ3 / RQ4`

也就是说，当前最小实验**不应该**优先追求：

- cross-benchmark transfer
- category-level calibration 的完整结论
- deployment cost / latency frontier

这些方向重要，但都不该排在“benign 学习是否真的生效”之前。

## 五、当前最小实验的建议通过标准

结合已有 run、`Discussion.md` 和当前问题，建议把当前最小实验的通过标准写成下面这组。

### A. 硬性通过条件

1. `Benign Learned > 0`
2. `TN > 0`
3. benign 样本不再全部落入统一拦截
4. `unsafe recall` 相对当前基线下降不超过 `5%`

### B. 优化有效的判据

1. `FP` 相比当前可比 run 有明确下降
2. `specificity` 相比当前基线有明确提升
3. `Skip Rate` 不继续恶化，最好下降
4. `benign_learned` 的增加伴随 `TN` 上升，而不只是日志计数增加

### C. 暂不作为本轮 blocker 的项目

1. cross-benchmark transfer
2. 完整 residual taxonomy 收尾
3. cost / latency benchmark
4. 分类别阈值的系统性校准

---

## 六、最终归纳：当前最小实验应该优化成什么样

如果把当前最小实验的目标压缩成一句话，可以写成：

**把最小实验从“unsafe 召回高但 benign 误杀严重、且 benign 覆盖还不稳定”的状态，优化到“benign 已被稳定学习、误杀开始明显下降、同时 unsafe 防御基本不掉”的状态。**

更落地一点，当前最小实验真正要回答的是这 3 个问题：

1. benign 样本现在是不是已经真的进入训练并触发学习？
2. 这些 benign 学习有没有真实降低 `FP`、提升 `TN / specificity`？
3. 在做到上面两点时，`unsafe recall` 是否仍然守住？

只要这三个问题还没有被清楚回答，当前最小实验的优化重点就不应该转移到更远的议题上。
